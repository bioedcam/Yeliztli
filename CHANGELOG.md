### Added

- **Sample-merge service core (Step 65 / MRG-02; Plan §10.2, §10.5):** New `backend/services/sample_merge.py::merge_samples(registry, source_sample_ids, individual_id, strategy, display_name) -> int` materialises a merged per-sample DB from two source samples linked to the same `individuals` row, returning the new `samples.id`. Pipeline (Plan §10.5 steps 1–8): (1) validation — `source_sample_ids` must list exactly two distinct ids; both must resolve to a `samples` row whose `individual_id` matches the request; both samples' most-recent `jobs` row with `job_type='annotation'` must carry `status='complete'`; neither may be stale per `backend.services.staleness.is_sample_stale` — membership / status / shape failures raise `InvalidMergeRequestError` (the Step 68 route maps to HTTP 422); stale-source raises `StaleSourceError` carrying `{error: 'stale_source_sample', stale_sample_ids: [...], message, reannotate_url}` (the Step 68 route maps to HTTP 423, same shape Plan §7.5 declares for `require_fresh_sample`); (2) open both source DBs read-only through `registry.get_sample_engine` and stream `raw_variants` into a `(chrom, pos)`-keyed in-memory map (~840 k loci × ~80 bytes ≈ 80 MB peak per source, acceptable per Plan §10.5 step 3); (3) collect the conflict-rsid subset (loci where both sources are present with different rsids — typically <1% of total loci) and probe `vep_annotations.rsid` via `registry.vep_engine` exactly once per probe batch (`_VEP_BUNDLE_PROBE_BATCH = 500`, under SQLite's 999-variable limit); a missing or unreachable bundle file logs `merge_vep_bundle_unreachable` and falls through to the §10.2 step 2 "both or neither in bundle → prefer S1" branch, so merge keeps working on machines that haven't installed the ~600 MB Step-4 bundle yet; (4) apply §10.2 / §10.3 semantics in `_apply_semantics`: unique-to-S1 → `(source='S1', concordance='unique', alt_rsid='')`, unique-to-S2 → symmetric; same coordinate both sides → rsid-collapse tiebreaker first (bundle-hit wins, then S1 fallback, with the losing rsid written to `alt_rsid` and `concordance_summary.collapsed_rsid` incremented as an additive marker per Plan §10.4c), then genotype concordance — both no-call → `match` (round-trips through `is_no_call` so downstream findings stay suppressed), exactly one no-call → `filled_nocall` with `source` pointing at the calling side, both called and equal → `match` with `source='both'`, both called and discordant → strategy decides (`FLAG_ONLY` writes `genotype='??'` and `discordant_alt_genotype='S1=<s1_gt>;S2=<s2_gt>'`; `PREFER_23ANDME` / `PREFER_ANCESTRYDNA` resolve the winner by reading each sample's `file_format` prefix and write `discordant_alt_genotype='S<loser>=<loser_gt>'`). Output rows are sorted `(chrom, pos)` so test assertions are deterministic and bulk inserts land in PK order; (5) compute `file_hash = SHA-256(S1.file_hash | S2.file_hash | strategy.value | SAMPLE_SCHEMA_VERSION)` per Plan §10.5 step 5 — order-sensitive (rsid tiebreaker prefers S1) so `[A,B]` and `[B,A]` yield distinct hashes, and `SAMPLE_SCHEMA_VERSION` is included so a re-merge after a future schema bump produces a distinct hash; the helper `_compute_file_hash` is exposed for the Step 77 re-merge / hash-invariants test; (6) insert the new `samples` row with `file_format='merged_v1'`, `individual_id` propagated, allocate the id, then update `db_path = f"samples/sample_{id}.db"`; (7) **pre-create** the merged sample DB on a throwaway `sa.create_engine(...)` (not the registry's cached engine) so `create_sample_tables(bootstrap_engine, is_merged_sample=True)` from Step 64 materialises `raw_variants` with the `(chrom, pos)` PK *before* `registry.get_sample_engine` is invoked — otherwise the registry's `ensure_sample_schema_current` would materialise the default rsid-PK layout first and collide with the merged DDL; (8) write `sample_metadata` (single-row), the single-row `merge_provenance` (`{id=1, strategy, source_sample_ids=JSON [s1, s2], source_file_hashes=JSON, concordance_summary=JSON {match, filled_nocall, discordant, unique_S1, unique_S2, collapsed_rsid}}` enforcing Plan §10.4c's row-count invariant: `match + filled_nocall + discordant + unique_S1 + unique_S2 == total merged loci`), and bulk-insert `raw_variants` in 10 k-row batches matching the file-ingest path; (9) lazy-import `backend.tasks.huey_tasks.{create_annotation_job, run_annotation_task}` and enqueue the standard annotation job against `new_sample_id` — the caller polls existing `GET /api/annotation/status/{job_id}`. An enqueue failure is logged (`merge_annotation_enqueue_failed`) but never rolls back the merge, since the merged sample is intact on disk and the user can retry via `POST /api/annotation/{sample_id}`. Public surface: `merge_samples`, `MergeStrategy` (StrEnum: `PREFER_23ANDME`, `PREFER_ANCESTRYDNA`, `FLAG_ONLY`), `MergeError` / `InvalidMergeRequestError` / `StaleSourceError`, `_compute_file_hash` (exposed for test parity with the Plan §10.5 step 5 recipe). - **`is_merged_sample` factory parameter on `create_sample_tables` (Step 64 / MRG-01 part 2; Plan §10.4a):** `backend/db/sample_schema.py::create_sample_tables` grows a keyword-only `is_merged_sample: bool = False` argument that selects the `raw_variants` primary-key layout at table-creation time. Default (`False`) keeps the historical `rsid` PK every single-vendor sample DB has used since v1 — covers regular file ingest (`backend/api/routes/ingest.py:156`), every test fixture (`tests/backend/conftest.py:48` `sample_engine`, `test_haplogroup.py`, `test_stale_sample_dependency.py`), and every other pre-Phase-3 caller without modification. Passing `is_merged_sample=True` (only `backend/services/sample_merge.py` will do this, landing in Step 65) materialises `raw_variants` with a composite `PRIMARY KEY (chrom, pos)` because the canonical merge key from Plan §10.2 is the coordinate pair (two source samples carrying different rsids at the same `(chrom, pos)` collapse to one row; rsid would have produced an artificial dup-or-drop). The four v8 provenance columns (`source`, `concordance`, `discordant_alt_genotype`, `alt_rsid`) ship in both DDL paths so a downstream `INSERT` doesn't have to branch by sample type. Implementation strategy: rather than the Plan-§10.4a "fresh `MetaData()` per call" sketch (which would force redefining 13+ unrelated tables per sample-DB open), the merged-PK branch issues a single raw `CREATE TABLE raw_variants ... PRIMARY KEY (chrom, pos)` DDL *before* `sample_metadata_obj.create_all(engine, checkfirst=True)` runs — `checkfirst=True` then sees `raw_variants` already exists and skips it, while still creating every other table (`annotated_variants`, `merge_provenance`, `annotation_state`, `tags`, `watched_variants`, `findings`, `apoe_gate`, `haplogroup_assignments`, `lai_results`, `panel_coverage`, `variant_tags`, `variant_overlays`) from the module-level metadata. The redundant `idx_raw_chrom_pos` index that `create_all` also issues against the (already PK-indexed) `(chrom, pos)` pair is harmless on SQLite (verified by an inspector smoke test — the explicit index coexists with the autoindex for the PK). Per Plan §10.4a invariant, only `raw_variants` swaps its PK — `annotated_variants` keeps `rsid` PK (annotation engine upserts via `index_elements=["rsid"]` and merged samples carry one rsid per locus from the §10.2 collapse, so writes succeed unchanged), `variant_tags` keeps composite `(rsid, tag_id)`, `panel_coverage` keeps `(module, rsid)`, and `watched_variants` keeps `rsid`. The v7→v8 in-place upgrade path (`ensure_sample_schema_current` → `_add_missing_columns`) is intentionally unchanged: Plan §10.4 final paragraph locks "The (chrom, pos) PK divergence does not apply to in-place v7→v8 upgrades" — every existing single-vendor DB keeps rsid PK forever, the new PK only takes effect on freshly-created merged-sample DBs that the Step 65 merge service produces. Standard readers continue to work transparently against either layout because the module-level `raw_variants` Table object is referenced for its columns (`raw_variants.c.rsid`, `.c.chrom`, `.c.pos`, `.c.genotype`) in `sa.select(...)` calls throughout `backend/annotation/{clinvar,vep_bundle,dbsnp,vcfanno_runner}.py` and `backend/ingestion/vcf_export.py` — PK is a constraint, not a query requirement, so the select round-trips identical rows from both PK variants (locked by `test_standard_reader_returns_rows_from_both`). Tested in `tests/backend/test_sample_schema_migration_v7_v8.py::TestIsMergedSampleFactory` (9 cases, all green): default omitting the flag → rsid PK; explicit `is_merged_sample=False` → rsid PK; `is_merged_sample=True` → `(chrom, pos)` PK; merged DB carries the full v8 surface (four provenance columns + `merge_provenance` table + `annotated_variants` still rsid-PK as the negative cross-table assertion); merged DB stamped at `SAMPLE_SCHEMA_VERSION == 8`; PK swap proven from both sides — merged DB rejects two different-rsid rows at the same `(chrom, pos)` with `IntegrityError` (locks the new PK fires) and accepts two same-rsid rows at different coordinates (would have failed under the old rsid PK; locks the swap is real, not just additive); unmerged-default negative-control still rejects duplicate-rsid inserts (regression boundary — `is_merged_sample=False` must not silently lift the old constraint); and the standard-reader round-trip across both PK variants returns byte-identical row sets via the module-level `raw_variants` Table import. Total file is now 23 cases up from 16. Per CLAUDE.md SOP #8 the cross-cutting full-sweep gate ran (`pytest tests/backend/` — flagged ⚠ full-sweep because the `raw_variants` PK branches on a factory parameter and every reader / writer must still work both ways). Risk register R-15 (data-integrity) is the load-bearing item: the PK swap is the storage layer the Step 65 merge service writes into, and the constraint-not-query-requirement contract — proven by the byte-identical select round-trip — keeps every annotation reader untouched. R-12 (genotype concordance) is the secondary item: the `(chrom, pos)` PK is what makes a different-rsid-same-coordinate collapse a hard rejection at INSERT time rather than a silent dedup-or-shadow at SELECT time, surfacing inconsistent source-sample provenance to the merge service explicitly. **Decisions**

- **Coord-fallback wiring lives in the engine, not in `_lookup_vep`.** The four concurrent annotation-source adapters (`_lookup_vep`, `_lookup_clinvar`, `_lookup_gnomad`, `_lookup_dbnsfp`) each return a plain `dict[str, dict]`; adding internal coord-fallback to `_lookup_vep` would either change its return signature (breaking `TestLookupVep` and other consumers) or hide the rsid-vs-coord distinction the Plan §5.6 telemetry requires. The cleanest factoring mirrors how `_lookup_gene_phenotype` is already plumbed: run the new helper sequentially in the per-batch loop *after* `as_completed` populates `vep_data`, then continue into gene-phenotype lookup so it sees both rsid and coord-matched VEP rows. `_lookup_gnomad` / `_lookup_dbnsfp` already use raw_by_rsid for their own coord fallbacks; the `_lookup_vep` adapter accepted `raw_by_rsid` as an unused parameter and now stays consistent with that interface while the coord-fallback step uses it in the caller.
- **The in-memory bundle, not the on-disk `mini_vep_bundle.db`.** Step 40's spec reads "using the regenerated mini bundle"; the test file uses the same `vep_seed.csv` source the regen script consumes (so it cannot drift from the mini bundle's rsID coverage) and adds two extra rows at the exact `(chrom, pos)` of the fixture's kgp\* entries. This is the same in-memory pattern `test_annotation_engine.py::vep_engine_inmemory` already uses for the existing coverage_stats tests; introducing a separate on-disk-bundle path in the new file would require either a writable copy of `mini_vep_bundle.db` or a temp-dir regenerate-on-fixture-startup step, neither of which step 40 mandates. The two coord-fallback rows carry the `rs_coord_fallback_{0,1}` rsIDs (deliberately non-`kgp*`) and a placeholder `FIXTURE_GENE` / `intron_variant` payload — sufficient for the bitmask + telemetry-shape assertions; the per-row annotation content is not inspected.
- **gnomAD / dbNSFP intentionally left unavailable in the fixture.** The ADNA-09 contract is VEP-bundle hit-rate; the mock registry's `gnomad_engine` / `dbnsfp_engine` properties raise `RuntimeError` so the engine takes the `_check_engine_available → None` branch for each, exercising the graceful-degradation path on a fresh AncestryDNA install where the optional databases have not yet been downloaded. The engine still emits the full coverage_stats payload because that path runs unconditionally at the end of `run_annotation`.

#### Step 30 — AncestryDNA parser

##### Added

- New `backend/ingestion/parser_ancestrydna.py` (ADNA-04; Plan §8.5, §8.6) — implements the AncestryDNA V2.0 raw-data TSV parser onto the shared `backend.ingestion.base` types. Public entry point is `parse_ancestrydna(file_or_path)` returning a `base.ParseResult` with `vendor=SourceVendor.ANCESTRYDNA`, `version="v2.0"` (string per Plan §8.7), `build="GRCh37"`. The dispatcher's lazy-import branch in `backend/ingestion/dispatcher.py::parse` (introduced in step 27, gated on this module's existence) now routes AncestryDNA inputs end-to-end without raising `UnsupportedFormatError`. Vendor dispatch (binary rejection, head-line precedence, VCF/CSV/unknown guidance) stays exclusively in the dispatcher — this module assumes its caller has already identified the file as AncestryDNA.
- **`detect_version(comment_lines, has_uncommented_5col_header)`** (Plan §8.5) — signature-driven version resolution: (i) an explicit `AncestryDNA array version: V<X>` comment wins (regex `array\s+version:\s*V([0-9.]+)`, case-insensitive); (ii) the `#AncestryDNA` vendor signature plus an uncommented 5-column header fall back to `"v2.0"` (covers real exports, which usually omit the explicit array-version line); (iii) `"unknown"` only when neither signal is present. Unknown bubbles up as `UnrecognizedVersionError` in `parse_ancestrydna`, never a silent success.
- **`_canonical_genotype(a1, a2)`** — parse-time canonicalization per Plan §8.6 #1, #2, #6, #7: element-wise no-call sentinels `{"0", ""}` collapse the pair to `"--"` (so `A`/`0` → `"--"`, not `"A-"`); otherwise both alleles uppercase and the pair sorts to absorb the `I<TAB>D` vs `D<TAB>I` indel-ordering noise seen in real exports (canonical forms: `"DD"`, `"DI"`, `"II"`). Hemizygous chrX / chrY calls land as `"AA"` / `"CC"` etc. via the same path — the downstream zygosity classifier handles haploid-homozygous (Plan §8.6 #3).
- **`_validate_line(parts, line_num)`** — single-row validator: enforces 5 columns, non-empty rsid, integer position ≥ 0, and chromosome normalization via `backend.ingestion.chromosomes.normalize_for(SourceVendor.ANCESTRYDNA, chrom_raw)` (which performs the PAR `25 → X` collapse and the `26 → MT` map per Plan §8.4). `MalformedDataError` carries the raw offending value in `!r`-quoted form for log-line forensics, matching the 23andMe parser's style. Legacy `kgp*` IDs (1000 Genomes proxies) pass through verbatim per Plan §8.5 — the rsid string is not rewritten, the downstream VEP-bundle rsid index misses them, and the coordinate-fallback path picks them up.
- **IO contract** mirrors `parser_23andme`: `_open_input(file_or_path)` accepts a `str` / `pathlib.Path` / open `TextIO` and uses `encoding="utf-8", errors="replace"` so stray non-UTF-8 bytes in comment lines surface as replacement characters rather than aborting the parse (Plan §8.6 #9 — locked by `test_non_utf8_byte_in_comment_replaced`). `_read_head_lines` rewinds seekable streams so detection does not consume the body. `_detect_format` scans the first 50 lines collecting comments + checking for the uncommented 5-column header, then delegates to `detect_version`. CRLF (`\r\n`) line endings are stripped per-line so Windows-exported files parse identically to LF (Plan §8.6 #8 — locked by `test_crlf_line_endings`). Trailing blank lines are tolerated and counted in `skipped_lines` (Plan §8.6 #5).
